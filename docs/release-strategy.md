# Release Strategy

This repository is a documentation project, so releases should communicate meaningful knowledge and workflow milestones.

## What Counts as a Release

Create a release when one or more of the following happens:

- substantial chapter additions or rewrites
- major structure/navigation improvements
- contributor workflow or maintainer process upgrades
- publishing pipeline improvements that affect reliability

Avoid publishing releases for isolated typo fixes unless bundled into a broader update.

## Milestone Release Cadence

Recommended cadence:

- milestone release every 4 to 8 weeks when enough meaningful changes accumulate
- optional patch release for urgent workflow or security-related fixes

## Release Content Structure

Each release should include:

1. Reader highlights
- chapter additions
- improved explanations
- new references or drills

2. Contributor highlights
- onboarding improvements
- issue template or review process updates

3. Maintenance highlights
- CI/workflow reliability improvements
- metadata/discoverability upgrades

## Writing Release Highlights People Read

Use concise, concrete bullets:

- what changed
- why it matters
- where to start (link to one or two key pages)

Avoid vague release text such as general improvements across the board.

## Using GitHub Releases to Build Momentum

- Keep releases regular enough to show project activity.
- Link each release to roadmap progression in docs/oss-growth-roadmap.md.
- Include one clear call to action for readers and one for contributors.

## Suggested Call-to-Action Patterns

- Reader CTA: Start with docs/getting-started.md and Part 1.
- Contributor CTA: Pick one issue from docs/good-first-issues.md guidance.
- Maintainer CTA: Apply manual settings from docs/manual-github-settings-checklist.md.
